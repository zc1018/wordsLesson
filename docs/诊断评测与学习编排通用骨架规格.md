# 诊断评测与学习编排通用骨架规格

版本：v0.1  
日期：2026-07-06  
适用范围：100天核心5000词，以及后续可迁移到阅读、写作、数学、专业课等营类产品的诊断与编排系统  
定位：产品委员会 / OPE / 教研 / 技术 / 运营共同使用的规格骨架

## 一、总定义

### 1. 一句话定义

建立一套可复用的“诊断评测 + 学习对象打标 + 个性化编排 + 过程校准 + 结营报告”骨架，让营类产品不只是给内容，而是能识别用户起点、动态安排学习路径，并在结营时证明变化。

### 2. 核心闭环

```text
能力地图
  -> 入门诊断
  -> 学习对象打标
  -> 个性化学习顺序
  -> 学习与练习事件回流
  -> 阶段诊断与重新编排
  -> 结营复诊与报告
  -> 下一步产品推荐
```

### 3. 设计原则

- 诊断不是单次考试，而是学习路径的入口。
- 打标不是后台统计，而是编排系统的最小决策单位。
- 编排不是简单排序，而是资源分配：哪些先学、哪些轻触达、哪些重点纠偏、哪些进入复习队列。
- 结营报告不是总结页，而是让用户看见变化、触发续转或转介绍的交付物。
- 系统必须抽象成跨科目能力，而不是只服务词汇一个产品。

## 二、系统能力分层

### 1. L0：产品配置层

用于定义一个营类产品的基础骨架。

| 模块 | 说明 |
| --- | --- |
| 产品周期 | 例如 7 天、20 天、100 天 |
| 能力地图 | 该产品要提升的能力维度 |
| 学习对象 | 词、题型、知识点、文章、题组、任务等 |
| 内容资源 | 课程、讲义、题目、视频、练习、报告模板 |
| 诊断节点 | 入门测、阶段测、结营测 |
| 编排规则 | 学习顺序、复习规则、跳过规则、纠偏规则 |
| 报告模板 | 学生报告、运营报告、教研报告 |

### 2. L1：诊断评测层

负责回答三个问题：

1. 用户当前在哪个水平。
2. 用户在哪些能力维度或知识对象上薄弱。
3. 哪些学习对象应优先进入学习、复习或纠偏。

诊断类型：

| 类型 | 触发时间 | 目的 |
| --- | --- | --- |
| 入门诊断 | 学习前 | 建立起点基准和初始标签 |
| 阶段诊断 | 学习中 | 校准掌握情况，调整后续路径 |
| 结营诊断 | 学习结束 | 对比起点，证明变化，推荐下一步 |
| 过程诊断 | 每日练习中 | 根据行为事件实时微调 |

### 3. L2：学习对象打标层

负责给每个学习对象生成状态。

示例：

| 科目 | 学习对象 |
| --- | --- |
| 词汇 | 单词、词族、词根、搭配、易混组 |
| 阅读 | 文章、题型、长难句、段落功能、定位线索 |
| 写作 | 句型、论证结构、常见错误、素材表达 |
| 数学 | 知识点、题型、公式、方法、易错点 |
| 专业课 | 知识点、章节、概念、案例、论述题模板 |

统一标签：

| 标签 | 含义 | 编排动作 |
| --- | --- | --- |
| mastered | 稳定掌握 | 轻触达、低频复习 |
| partial | 半熟或不稳定 | 正常学习、早期复习 |
| weak | 薄弱 | 完整学习、增加练习 |
| unknown | 未掌握 | 优先学习、重点复习 |
| false_familiar | 虚熟或误认 | 先纠偏再学习 |
| high_value | 高价值对象 | 即使掌握也可周期回看 |
| high_risk | 高错率/易混/易忘 | 提高复习权重 |

### 4. L3：学习编排层

负责根据标签和产品目标生成每日任务。

编排结果不是一个静态课表，而是一组任务队列：

| 队列 | 用途 |
| --- | --- |
| new_learning_queue | 新学内容 |
| review_queue | 复习内容 |
| correction_queue | 错题/虚熟/误认纠偏 |
| light_touch_queue | 已掌握内容的轻触达 |
| assessment_queue | 阶段诊断题或结营测题 |
| recommendation_queue | 结营后的后续产品推荐 |

### 5. L4：报告与推荐层

报告需要同时服务三类人：

| 对象 | 报告目的 |
| --- | --- |
| 学员 | 看懂起点、过程和变化 |
| 运营 | 跟踪留存、完课、续转机会 |
| 教研/产品 | 判断内容和题目是否有效 |

报告不是算法展示，应表达为：

- 你现在在哪里。
- 你哪里变好了。
- 你哪里还薄弱。
- 下一步应该怎么学。
- 为什么推荐这个后续产品。

## 三、数据模型骨架

| 字段 | 说明 |
| --- | --- |
| program_id | 产品 ID |
| program_name | 产品名称 |
| subject | 科目 |
| duration_days | 周期 |
| objective | 产品目标 |
| version | 配置版本 |
| status | draft / active / archived |

### 2. ability_dimension

定义能力维度。

| 字段 | 说明 |
| --- | --- |
| dimension_id | 能力维度 ID |
| program_id | 所属产品 |
| name | 维度名称 |
| parent_id | 上级维度 |
| weight | 权重 |
| description | 说明 |

词汇产品示例：

- 词义识别
- 词频层级掌握
- 词根词缀意识
- 词源/语义迁移理解
- 搭配识别
- 易混词辨析

数学产品示例：

- 概念理解
- 公式调用
- 题型识别
- 解题步骤
- 易错规避
- 综合应用

### 3. learning_object

定义可学习、可测评、可打标的最小对象。

| 字段 | 说明 |
| --- | --- |
| object_id | 学习对象 ID |
| program_id | 所属产品 |
| object_type | word / question_type / concept / passage / task |
| title | 名称 |
| unit_id | 所属单元 |
| difficulty | 难度 |
| importance | 重要性 |
| tags | 方法、题型、场景、考试标签等 |
| prerequisite_ids | 前置对象 |
| content_refs | 对应课程/讲义/题目/视频 |

### 4. diagnostic_item

定义测评题或诊断任务。

| 字段 | 说明 |
| --- | --- |
| item_id | 题目 ID |
| program_id | 所属产品 |
| object_id | 关联学习对象，可为空 |
| dimension_ids | 测量哪些能力 |
| item_type | yes_no / mcq / matching / short_answer / task |
| difficulty | 难度 |
| discrimination | 区分度，MVP 可人工标注 |
| answer_key | 标准答案 |
| scoring_rule_id | 评分规则 |
| is_control_item | 是否校准项 |

### 5. user_object_state

每个用户对每个学习对象的状态。

| 字段 | 说明 |
| --- | --- |
| user_id | 用户 ID |
| program_id | 产品 ID |
| object_id | 学习对象 ID |
| mastery_score | 0-100 |
| mastery_label | mastered / partial / weak / unknown / false_familiar |
| evidence_type | direct_calibrated / direct_response / inferred / practice_updated / stage_test |
| confidence | high / medium / low |
| priority | low_review / normal / high / correction |
| next_action | light_touch / normal_learning / full_learning / correction |
| review_weight | 复习权重 |
| mistake_type | none / recall_error / concept_error / confusion / false_familiar |
| last_updated_at | 更新时间 |

### 6. learning_event

记录用户学习过程。

| 字段 | 说明 |
| --- | --- |
| event_id | 事件 ID |
| user_id | 用户 ID |
| program_id | 产品 ID |
| object_id | 学习对象 ID |
| task_id | 任务 ID |
| event_type | view / answer / complete / skip / retry |
| response | 用户作答 |
| is_correct | 是否正确 |
| response_time_ms | 反应时 |
| attempt_count | 尝试次数 |
| created_at | 时间 |

### 7. report_snapshot

固化报告快照，避免后续数据更新导致历史报告漂移。

| 字段 | 说明 |
| --- | --- |
| report_id | 报告 ID |
| user_id | 用户 ID |
| program_id | 产品 ID |
| report_type | entry / stage / final |
| period | D0 / D20 / D40 / D60 / D80 / D100 |
| summary_json | 学生端摘要 |
| metrics_json | 指标数据 |
| recommendations_json | 后续建议 |
| created_at | 生成时间 |

## 四、核心流程规格

### Flow 1：产品初始化

目标：把一个科目的营类产品配置成可诊断、可编排、可报告的系统。

输入：

- 产品周期。
- 能力地图。
- 学习对象清单。
- 内容资源。
- 题库或任务库。
- 编排规则。
- 报告模板。

输出：

- program 配置。
- ability_dimension 配置。
- learning_object 图谱。
- diagnostic_item 题库。
- scoring_rule 评分规则。
- orchestration_rule 编排规则。

验收：

- 每个学习对象至少归属一个能力维度。
- 每个学习对象有 difficulty、importance、unit_id。
- 每个诊断节点有可抽题范围和报告模板。
- 至少有一条从标签到学习动作的映射规则。

### Flow 2：入门诊断

目标：建立用户起点，生成初始对象状态。

通用流程：

1. 展示测评说明。
2. 完成快速筛查或基础题。
3. 完成含义/能力校准题。
4. 记录反应时、正确率、稳定性。
5. 计算能力维度分。
6. 给全部学习对象生成初始状态。
7. 输出起点报告。
8. 生成第一阶段学习路径。

词汇产品实例：

- A 段：1K-5K 分层抽样快速识别。
- B 段：含义选择题校准。
- 控制项：识别虚熟或过度自信。
- 输出：5000 个词的熟悉度标签。

可复用抽象：

| 词汇 | 其他科目 |
| --- | --- |
| 单词熟悉度 | 知识点掌握度 |
| 词频层级 | 难度层级 |
| 词义校准 | 概念/题型校准 |
| 虚熟词 | 假会题/误解概念 |
| 复习权重 | 练习权重 |

### Flow 3：学习对象打标

目标：将诊断结果转化成可执行编排状态。

评分来源优先级：

1. 直接测得且有校准题：高置信度。
2. 直接作答但未校准：中置信度。
3. 同类对象推断：低置信度。
4. 后续练习多次验证：高置信度。

标签生成规则：

| 条件 | 标签 | 动作 |
| --- | --- | --- |
| 答对且反应快 | mastered | 轻触达 |
| 答对但反应慢 | partial | 正常学习 |
| 自评会但校准错 | false_familiar | 重点纠偏 |
| 答错且低层级 | weak | 完整学习 |
| 未测且同类低掌握 | unknown / weak | 完整学习 |
| 未测且同类高掌握 | inferred_partial / mastered | 低置信轻触达 |

关键约束：

- 未直接测得对象不能标记为“确定掌握”，只能给推定标签和低置信度。
- 每个对象必须保留 evidence_type 和 confidence。
- 后续练习可以更新标签，但报告快照必须保留历史版本。

### Flow 4：学习顺序编排

目标：把对象状态转化成每日学习任务。

编排输入：

- 用户对象状态。
- 学习对象依赖关系。
- 每日认知负荷上限。
- 产品阶段目标。
- 复习间隔规则。
- 错误类型和优先级。

编排优先级：

1. 当天必须完成的主线对象。
2. 高价值且薄弱对象。
3. false_familiar / high_risk 对象。
4. 到期复习对象。
5. 前置依赖对象。
6. 低置信度推断对象的抽样验证。
7. 已掌握对象的轻触达。

每日任务结构：

| 模块 | 来源 |
| --- | --- |
| 今日主线 | 课程单元或能力目标 |
| 新学任务 | weak / unknown / normal 对象 |
| 轻触达 | mastered 或低置信推定掌握对象 |
| 纠偏任务 | false_familiar / 错误高发对象 |
| 复习任务 | 到期对象、错题对象 |
| 退出反馈 | 今日掌握变化和明日建议 |

词汇产品约束：

- 每日总触达词数不超过 70。
- 熟悉词可折叠或快速过。
- 虚熟词先纠偏再讲解。
- 错词自动提前进入复习。

跨科目约束示例：

| 科目 | 编排上限 |
| --- | --- |
| 阅读 | 每日文章/段落数量、题型数量、长难句数量 |
| 写作 | 每日句型、错误类型、改写任务数量 |
| 数学 | 每日知识点、例题、练习题、错题回炉数量 |
| 专业课 | 每日概念、章节、案例、背诵任务数量 |

### Flow 5：过程数据回流

目标：让学习行为不断修正标签，而不是只依赖入门测。

事件更新规则示例：

| 行为 | 更新 |
| --- | --- |
| 答对且反应快 | mastery_score +6 |
| 答对但反应慢 | mastery_score +3 |
| 课后复习答对 | mastery_score +4 |
| 阶段测答对 | mastery_score +8 |
| 答错 | mastery_score -10 |
| 连续两次纠正虚熟 | 移出 false_familiar |
| 长期未复现 | 自然衰减 5-10 |

跨科目通用做法：

- 正确率只是一部分。
- 反应时、犹豫、重试、跳过都应进入状态更新。
- 用户觉得会但做错，是比“不会”更重要的风险信号。

### Flow 6：阶段诊断与重新编排

目标：每个阶段给用户反馈，同时调整后续路径。

阶段节点：

- 词汇 100 天产品：D20 / D40 / D60 / D80 / D100。
- 7 天营：D0 / D3 / D7。
- 20 天营：D0 / D10 / D20。

阶段测输出：

- 当前阶段掌握率。
- 相比入门测或上一阶段的变化。
- 错误类型分布。
- 薄弱能力维度。
- 需要提前复习或延后学习的对象。
- 下一阶段编排调整。

重新编排规则：

| 信号 | 动作 |
| --- | --- |
| 某维度持续低于阈值 | 增加该维度任务比例 |
| 某对象多次错误 | 进入纠偏队列 |
| 某对象长期稳定正确 | 降低复习频率 |
| 阶段测暴露新薄弱点 | 插入补救单元 |
| 用户进度落后 | 降低新学量，提高轻触达比例 |

### Flow 7：结营复诊与报告

目标：证明变化，给出下一步路径。

结营报告模块：

1. 起点与终点对比。
2. 能力维度变化。
3. 学习对象掌握变化。
4. 错误类型减少情况。
5. 高价值成就证据。
6. 仍然薄弱的能力。
7. 后续学习建议。
8. 可承接产品推荐。

词汇产品报告示例：

- 入门测熟悉词约 X 个，结营测约 Y 个。
- 1K-5K 各层级掌握率变化。
- 虚熟词减少数量。
- 高频错词减少数量。
- 词根词缀/词源方法掌握变化。
- 后续推荐：精读、写作、口语、专项考试营。

跨科目报告模板：

| 报告模块 | 词汇 | 数学 | 写作 | 专业课 |
| --- | --- | --- | --- | --- |
| 起点 | 词汇底盘 | 知识点掌握 | 写作错误画像 | 章节掌握 |
| 过程 | 复习/错词 | 题型练习 | 改写/范文 | 背诵/应用 |
| 终点 | 结营测 | 综合测 | 终稿评分 | 模拟题/口述 |
| 变化 | 熟悉度提升 | 正确率提升 | 错误减少 | 召回率提升 |
| 推荐 | 阅读/写作等 | 强化题型 | 高阶写作 | 专项冲刺 |

## 五、抽象复用规范

### 1. 通用概念映射

| 通用概念 | 词汇产品 | 阅读产品 | 写作产品 | 数学产品 | 专业课产品 |
| --- | --- | --- | --- | --- | --- |
| learning_object | 单词 | 文章/题型/长难句 | 句型/错误/结构 | 知识点/题型 | 概念/章节/论述题 |
| ability_dimension | 词义/词根/搭配 | 定位/推断/结构 | 论证/语法/表达 | 概念/公式/步骤 | 理解/记忆/应用 |
| diagnostic_item | 词义题 | 阅读题 | 改错/改写 | 例题/小题 | 概念题/简答 |
| false_familiar | 虚熟词 | 看懂但选错 | 会背句型但用错 | 会公式但不会题 | 会背但不会解释 |
| review_queue | 复习词 | 错题文章/题型 | 错误句型 | 错题/公式 | 高频遗忘点 |
| final_report | 词汇变化 | 阅读能力变化 | 写作能力变化 | 解题能力变化 | 章节掌握变化 |

### 2. 每个新科目接入的最小配置

新科目要接入这套骨架，必须提供：

1. 能力维度表。
2. 学习对象表。
3. 诊断题或任务表。
4. 打标规则。
5. 编排规则。
6. 阶段节点。
7. 报告模板。
8. 后续推荐规则。

如果没有这些，不能声称“接入诊断与编排系统”。

### 3. 不能抽象掉的科目差异

通用系统只负责骨架，不替代教研判断。

不可抽象掉的部分：

- 各科目的能力维度定义。
- 题目评分标准。
- 学习对象之间的真实依赖关系。
- 结营报告里“提升”的教育解释。
- 后续产品推荐的业务边界。

## 六、产品页面骨架

### 1. 学员端页面

| 页面 | 核心内容 |
| --- | --- |

## 九、OpenSpec 风格需求清单

### Requirement 1：系统必须支持产品级能力地图

系统 SHALL 允许每个 program 配置多级 ability_dimension。

验收：

- 一个产品可以配置至少 2 层能力维度。
- 每个学习对象可关联一个或多个能力维度。
- 报告能按能力维度聚合结果。

### Requirement 2：系统必须支持入门诊断

系统 SHALL 在用户进入营后触发入门诊断，并生成起点报告。

验收：

- 诊断可在 8-12 分钟内完成。
- 能记录作答、正确率、反应时、控制项表现。
- 能生成用户对象状态。
- 能生成学生可读起点报告。

### Requirement 3：系统必须支持全量对象打标

系统 SHALL 为产品下所有 learning_object 生成用户级状态。

验收：

- 每个 user_object_state 有 mastery_score、mastery_label、evidence_type、confidence。
- 直接测得对象和推断对象必须区分。
- 编排系统能读取标签。

### Requirement 4：系统必须支持个性化编排

系统 SHALL 根据用户对象状态生成每日任务。

验收：

- 每日任务包含新学、复习、纠偏、轻触达。
- 可配置每日认知负荷上限。
- 错误高发和虚熟对象优先进入纠偏队列。
- 阶段测后能重新生成后续路径。

### Requirement 5：系统必须支持学习事件回流

系统 SHALL 将学习行为写入 learning_event，并更新对象状态。

验收：

- 答题正确率、反应时、重试、跳过可记录。
- 状态更新可追溯到事件。
- 虚熟对象纠正后可移出纠偏队列。

### Requirement 6：系统必须支持阶段报告

系统 SHALL 在配置节点生成阶段报告。

验收：

- 支持 D20/D40/D60/D80/D100 或其他周期配置。
- 报告显示相比上阶段变化。
- 报告能输出下一阶段编排建议。

### Requirement 7：系统必须支持结营报告与后续推荐

系统 SHALL 在结营时生成最终报告，并根据薄弱点推荐后续产品。

验收：

- 结营报告包含起点与终点对比。
- 推荐理由来自真实诊断结果。
- 推荐规则可配置，不硬编码到词汇产品。

### Requirement 8：系统必须跨科目复用

系统 SHALL 将能力维度、学习对象、诊断题、编排规则、报告模板抽象配置化。

验收：

- 至少能用同一数据结构配置词汇和数学两个产品。
- 不同科目的 scoring_rule 可独立实现。
- 报告模板可按科目替换。

## 十一、压力测试结论

### 1. 最大风险：把“通用骨架”做成空泛平台

建议答案：

先只允许一个主产品完整跑通，再用一个第二科目做复用验证。不要一开始为所有科目做大平台。

落地策略：

- 第一闭环：100天核心5000词。
- 第二验证：数学 7 天营或阅读营。
- 只有两个科目都能跑通，才抽象为平台能力。

### 2. 最大业务风险：报告看起来漂亮，但不能证明变化

建议答案：

每份报告必须绑定同一能力量尺和起终点对比。

防线：

- 入门测、阶段测、结营测共享能力维度。
- 报告展示“相比上次”的变化，而不是只展示当前分数。
- 不把未直接测得对象包装成确定掌握。

### 3. 最大技术风险：编排规则不可解释

建议答案：

MVP 必须规则优先，模型后置。

防线：

- 每个任务都有来源：新学、复习、纠偏、轻触达。
- 每个推荐都有原因。
- 每次状态更新能追溯事件。

### 4. 最大教研风险：标签不可信

建议答案：

每个标签必须有 evidence_type 和 confidence。

防线：

- 直接测得和推断分开。
- 校准题识别虚熟。
- 后续练习持续更新。
- 阶段测重新校准。

### 5. 最大产品风险：用户不愿意先测

建议答案：

入门测必须承诺明确收益，且 8-12 分钟内完成。

表达：

> 先测一下你的起点，测完后系统会少讲你已经会的内容，把时间留给真正薄弱的地方。

### 6. 最大复用风险：各科目差异被粗暴抹平

建议答案：

抽象流程，不抽象教研判断。

可复用：

- 能力地图。
- 对象打标。
- 编排队列。
- 报告框架。

不可复用：

- 评分细则。
- 能力解释。
- 题目质量标准。
- 教研干预策略。

## 十二、待决策问题

### Q1：第二个复用验证科目选什么？

推荐答案：

优先选数学营，而不是写作营。

原因：

- 数学也有明确学习对象、题型、错因和阶段测。
- 比写作更容易规则化验证通用骨架。
- 可以测试“非词汇对象”的抽象能力。

### Q2：入门测是否强制？

推荐答案：

对完整营强制，对体验课可跳过。

原因：

- 如果没有入门测，就没有个性化编排和结营对比。
- 但体验课强制测评会增加转化阻力。

### Q3：结营报告是否必须绑定后续推荐？

推荐答案：

必须绑定，但推荐理由必须来自报告薄弱点。

原因：

- 结营报告是续转接口。
- 不能把后续产品硬塞给所有人。

### Q4：通用骨架是否要首发后台配置？

推荐答案：

第一版不做完整后台，先用配置表 + 管理脚本 + 固定模板。

原因：

- 后台会显著拖慢首发。
- 先验证规则和闭环，再做配置界面。

## 十三、首版验收标准

词汇产品首版验收：

- 用户完成入门测后，系统生成 5000 个词的用户级标签。
- Day 1 学习顺序能读取标签并改变内容处理方式。
- 每日学习事件能更新标签。
- D20 阶段测能调整 D21-D40 的编排。
- D100 能生成结营报告。
- 结营报告能基于薄弱点推荐后续产品。

通用骨架验收：

- 同一套表结构能表达词汇和第二科目。
- 同一套流程能完成入门测、打标、编排、阶段报告、结营报告。
- 第二科目只替换能力地图、学习对象、题库、评分规则和报告模板，不改核心流程。

## 十四、当前最小下一步

建议下一步只做三个东西：

1. **词汇产品真实数据表**

2. **通用状态表和编排规则**
   - user_object_state。
   - learning_event。
   - orchestration_rule。

   - 跑一遍入门测到报告的闭环。

不要先做大后台，不要先做复杂模型，不要先做所有科目。
